# Day 031 - Enable

**Date:** 2026-07-22
**Pillar:** Enable

## Tidbit

Yesterday's ruling ended by admitting a rule does not write rows, so today is the part that gets it applied
by a person. The pattern is one I keep hitting in this repo: a control is only as good as the moment someone
actually runs it, and the cheapest place to protect an invariant is at the instant it is created, not in an
audit after. So this guide moves the whole handoff test to write time, to the two minutes before a row goes
down, where the only case that carries risk is a build that has been here before. The move I am keeping is
the fail-safe default. When a log-keeper cannot tell whether a returning build is the same handoff or a new
one, the guide tells them to merge, not split, because a wrongly-merged row is a quiet undercount by one and
a wrongly-split row hands a bounced build a fresh passing row and corrupts the send-back rate. Both are
errors, so you cannot avoid error under uncertainty, and the honest design picks which error you take. When
you cannot be right, be wrong in the direction that costs least.

## Shipped

`enablement/keeping-the-review-log.md` (v1): the write-time habit for whoever keeps the log, the people-side
of yesterday's ruling. It makes the point that a first-time build cannot be logged wrong in the way that
matters, so the whole risk lives in one reflex, opening a new row for a returning build whose last row
should have been updated, which is how the send-back-rate gaming gets in through the pen after the spec
closed it. The core is a four-step check to run before writing any row: has this build been here before (if
no, new row, done, which is nearly every row); if yes, does its last row have an open send-back; if so and
the submission answers it, same handoff, update and add a queue arrival; otherwise new handoff, new row. It
gives the fail-safe default (when unsure, merge not split, because the undercount is the smaller error), two
tells that land before the row is written (a name you have seen recently, a fix for a send-back you
remember), tied to Day 029's row 4. It closes honest: a guide makes the right call easy and cannot force it,
the log still rests on its named owner, and none of it helps a log that is not kept.
