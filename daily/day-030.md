# Day 030 - Govern

**Date:** 2026-07-21
**Pillar:** Govern

## Tidbit

The conflict yesterday's rows surfaced turned out to be one word doing two jobs. The queue used "handoff"
for a submission event, because a queue orders what is waiting right now and a returning build is genuinely
a new thing in line. The log and all three metrics used it for a review episode, because a log is a
permanent count and each trip should land in it once. Both readings are right for the record that uses
them, and giving two different units one name is what let the same event be written two ways. What decided
it was following the wrong reading to its consequence: if a re-submission opened a second row, a build sent
back and then approved would contribute a passing row of its own, so its rework would dilute the very
number that exists to count bounces. The metrics already ruled that out in writing; only the queue's wording
disagreed. The lesson I am keeping is that a definitional conflict is worth chasing to the number it
corrupts, because that is what tells you which reading to keep, and the fix was two names rather than a
better argument for one.

## Shipped

`governance/what-counts-as-one-handoff.md` (v1): the ruling that settles it. A **handoff** is one build's
trip through review, from the first time it is handed over to the decision that closes it, and the log
keeps one row per handoff; it is the unit all three review-side metrics count. An **arrival** is each time
a build enters the queue waiting on a decision, so one handoff can have more than one. The test for which
you are looking at: if the submission answers the reasons on an open send-back it is the same handoff, and
if it is new work or the handoff before it already closed on Approve or Approve with fixes it is a new one.
The practical version is that an open send-back is what keeps a handoff open. It argues the reading from
the gaming it prevents, quoting the send-back rate's own counts-once rule, and it is honest about the
price: review latency is a median over first reviews, so a re-submission's second wait attaches to no row,
which is a deliberate trade the live queue still sees. Both files now point here.
`sops/run-the-review-queue.md` goes to **v2** (a returning build is a fresh arrival, the waiting row's date
is named as its arrival time, ordering unchanged) and `sops/review-log-spec.md` to **v3** (the gap it named
yesterday is closed and it points at the ruling).
