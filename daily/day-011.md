# Day 011 - Document

**Date:** 2026-07-02
**Pillar:** Document

## Tidbit

I had two good documents this week, a builder self-check and a reviewer SOP, and still no document
owned the space between them. That space is the handoff, and it is exactly where work goes missing:
the builder assumes the reviewer knows what to run, the reviewer assumes the build arrived prepped,
and nobody wrote down what actually has to cross the line. Writing the connective SOP made me name
two things I had been leaving implicit. The handoff point, the single moment the build changes
hands, and the packet, the exact set of things that travel with it. It also made send-back a loop
instead of a shrug: the reasons come back in the same checklist, and the build re-enters the same
path rather than starting some new ad hoc thread. The lesson is that the seams between procedures
need documenting as much as the procedures do, because the cheapest place to drop something is a
boundary nobody owns.

## Shipped

`sops/hand-off-a-build-for-review.md` (v1): the connective SOP that ties the week's pieces into one
path. It names the handoff point, the single moment a build changes hands from builder to reviewer,
and defines the handoff packet: the runnable build, a one-paragraph description, and a confirmation
that the self-check is clean with an owner named and a rollback plan. The steps run from the builder
finishing the self-check (`enablement/builder-self-check.md`), through assembling the packet, handing
off, the reviewer running the review SOP (`sops/run-a-build-review.md`) and filling the checklist, to
the decision returning as that filled checklist. A send-back is a loop, not a dead end: the reasons
are the to-do list and the build re-enters the same path. It references the two guides rather than
copying them, so each stays the single source for its own side.
