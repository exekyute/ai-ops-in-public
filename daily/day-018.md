# Day 018 - Document

**Date:** 2026-07-09
**Pillar:** Document

## Tidbit

Two docs had been quietly leaning on a record that was never written down. The send-back rate said it
needed one row per handoff; the spot-audit said it needed the reviewer on that row; neither owned the
record itself. That is a specific kind of debt, shared infrastructure everyone references and nobody
defines, and it stays invisible until a second consumer needs the same thing shaped a little differently.
Writing the spec, the fields were the easy part. The load-bearing piece was one rule: one row per handoff,
first-review outcome frozen, and a re-review updates only the final outcome. Get that wrong and a rough
build could come around the loop, get approved the second time, and quietly erase its own send-back, which
is exactly the gaming the metric was built to resist. The lesson is that when two things read the same
record, the record needs its own doc and its own owner, or each consumer half-invents it and the two drift
apart.

## Shipped

`sops/review-log-spec.md` (v1): the review log, the record every review produces, so the send-back rate and
the spot-audit draw on one queryable log instead of scattered checklists. One row per handoff, four
required fields: build, reviewer, first-review outcome, date. The reviewer field is the one the spot-audit
named as a prerequisite it was waiting on, so this closes it. The load-bearing rule is that the first
review sets the row and a re-review updates only an optional final-outcome field, so the first-review
outcome stays frozen and the send-back rate keeps counting the first outcome (a sent-back-then-shipped
build cannot erase its own send-back). It draws the line between the log and the checklist: the checklist
is the full record of one review, the log is the queryable index across all of them that points back to
each. It names a named owner (per standard section 8), maps the three readers to their fields (send-back
rate = build, outcome, date; spot-audit = plus reviewer; a future review miss rate = the optional audit
fields), and is honest that it defines the record and not the tool, that the audit fields wait on a metric
not yet defined, and that a log only helps if reviews actually get logged.
