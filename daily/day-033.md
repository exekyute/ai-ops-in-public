# Day 033 - Analyze

**Date:** 2026-07-24
**Pillar:** Analyze

## Tidbit

The Analyze slot creates a quiet pressure to produce a number, and today the honest answer was that a
number is the wrong tool. Everything else in the analytics folder measures something nobody can read
directly: you cannot re-run last week's runs, cannot re-review a decided build, cannot know a wait that has
not finished. Each of those four carries a blind spot for exactly that reason. Index staleness is not like
that. The ground truth is sitting in the same directory, so every claim the index makes can be settled by
reading the tree again, which makes it decidable rather than estimable. Measure what you must estimate,
test what you can decide. The part I want to remember is what happened when I pushed on the argument
instead of accepting it: the check I thought was three rules turned out to be four, because the index opens
with a tally of its own that nothing was reading, and the rule that catches unindexed files immediately
flagged the memo writing it. A specification that finds its own next gap while being written is a better
sign than a clean one.

## Shipped

`analytics/measure-or-test.md` (v1): the memo arguing the index wants a check and not a fifth metric, and
the first usage-to-insight memo in a folder that until now held only the dictionary. It specifies what the
check reads: every cited path resolves, every version claim matches the file's H1 along with the title
beside it (which catches a pointer swapped to a different file at the same version), every in-scope file
appears in the index or in its deliberately-excluded list, and the opening tally recomputes. It takes the
metric idea seriously in two forms before rejecting it, the snapshot rate (whose only acceptable value is
100 percent, and which is a lossy summary of a result you already hold) and the process metric of how often
drift appears, which is a real future candidate that cannot exist until the check does and has a run
history. It sets that against the four existing metrics, each of which stands in for something unreadable
and is blind because of it. It is honest about the edges: rule 3 is mechanical only given a scope a person
decided, semantic drift and ordering need a person, and the check was run once by hand on Day 032 across 22
paths and 24 version claims with zero mismatches, which is a datapoint and not a habit. This repo has
shipped no code, so the check is specified and not implemented, and a specified check nobody runs catches
exactly as much as no check at all.
