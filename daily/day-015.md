# Day 015 - Build

**Date:** 2026-07-06
**Pillar:** Build

## Tidbit

Two weeks of this were all documents that describe a loop. None of them had ever shown the loop run.
That is a specific kind of gap: a set of pieces can each read as correct and still not prove they
connect, because nothing has carried one thing through all of them end to end. Writing the worked
example was the first time I made a build actually travel the path, and the useful move was to plant a
real flaw and let the loop catch it, rather than walk a clean build straight to approval. A build that
sails through proves nothing; the send-back is where you find out the seams actually carry a decision
back and the metric counts the right outcome. The lesson for week 3 is that documented and demonstrated
are different claims, and only the second one survives contact with a build.

## Shipped

`examples/a-build-through-the-loop.md` (v1): one synthetic build, `invoice-reminder-sheet-to-slack`,
run all the way around the review-and-handoff loop, from the bar it must meet to the number its review
produces. The build has one planted flaw, a Slack reminder that can double-post because the send is not
idempotent and carries no dedup key, and the builder marks the self-check's run-it-twice item as fine
by eye instead of running it. The trace follows the loop twice: the first review actually runs the day
twice, catches the double-post, and sends it back with one reason grounded in standard section 4; the
builder adds a dedup key, re-runs the self-check honestly, hands off again, and the second review
approves with a small wording fix. It closes by showing the number: this build adds one to the
send-back numerator because the metric counts the first-review outcome, not the final one, so a build
that eventually shipped still records that it arrived not ready. The honest close names what one
synthetic trip cannot prove: it shows the path exists, not that the path catches every kind of miss.
Adds a new `examples/` folder, with a row in the README layout.

## Next

Day 016 (Govern, Tue): the worked example exposed the loop's open assumption, that the review itself is
honest. A reviewer who eyeballs the same item the builder eyeballed approves the double-post and the
send-back rate never sees it. Start on a guard for that: a short reviewer-integrity check or a
spot-audit rule, so the number is not resting on trust alone.
