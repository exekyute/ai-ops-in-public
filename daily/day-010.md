# Day 010 - Enable

**Date:** 2026-07-01
**Pillar:** Enable

## Tidbit

A review gate catches problems, but catching a problem is the expensive place to fix it. By the
time a build reaches review, the description is missing and no owner is named, so the reviewer and
I trade messages for a day sorting out what should have been ready before I hit send. The checklist
is not wrong, it just lands on the reviewer, who is the person least able to fix any of it. Most of
those items are cheapest to close while the build is still in my hands. So today is the same quality
bar as yesterday's checklist, moved one step earlier and handed to the builder, who can act on it
fastest. That is the adoption idea from Day 003 in practice: put the check in the path of the work,
so doing it right is the easy choice and not an extra one. It does not replace review, a second
person still catches what I am blind to; it clears the obvious problems first so the review is
quick.

## Shipped

`enablement/builder-self-check.md` (v1): the builder's side of the v2 review checklist. Five short
groups of prepare-and-self-check actions, framed as what you do before handing a build off. Run it
yourself first (the happy path and a failure case). Make it safe to run (secrets in the store,
separate test and prod credentials, retries only where safe). Make it operable by someone else, the
part a reviewer cannot do for you: name one owner, set up a versioned definition and a safe stop as
the rollback plan, point alerts somewhere a person reads, and track volume and exception rate. The
AI-only checks if the build calls a model. And make it legible: the one-paragraph description,
sticky notes, an SOP if it runs in production. It cites the checklist and
`standards/automation-standards.md` so the two stay in sync, and it ends honest, that self-check is
preparation and not a substitute for a second reviewer.
