# Day 009 - Govern

**Date:** 2026-06-30
**Pillar:** Govern

## Tidbit

A standard and the checklist that enforces it have to move together, or the checklist quietly
starts approving builds the standard says no to. Yesterday I took the automation standard to v2 and
added two rules, who owns a live build and how you back a bad change out. For a day the checklist
did not know about them, and that gap is the easy place for a rule to die: it is written down, but
nothing at the gate asks for it. So today closes the gap I opened rather than breaking new ground,
plus two existing standard rules the v1 checklist had quietly skipped, the exception-rate metric
and the expected-failure split. The test of a check is still the same, two people should mark it
the same way without arguing about what it means.

## Shipped

`governance/ai-build-review-checklist.md` taken to **v2**. A new "Ownership and lifecycle" section
covers the standard's section 8 and section 9: a named owner recorded, alerts going somewhere a
person reads, the health metric read on a cadence, credentials rotated, explicit handover when the
owner changes, the definition versioned and revertible, a safe stop, and a change tested against
sample input before real data. Basics picked up the checks the v1 had skipped: separate test and
prod credentials with an env suffix, scheduled rotation with credential usage documented, and the
run-volume and exception-rate metric, with the exception rate defined to match
`analytics/metric-definitions.md` (runs that errored or fell out to a person). I also split two
checks that were doing double work, so retries cover the cap and backoff, and error handling
separates expected failures from unexpected ones, and added a sticky-note check to Documentation.
Every v1 check was kept; wording changed only where the standard changed. The end of the file lists
what changed.
